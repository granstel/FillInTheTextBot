## 3) Блокирующий этап: сначала интеграционные тесты (регресс)

Перед любым апдейтом зависимостей и фреймворка нужно зафиксировать текущее поведение.

### 3.1 Минимальный обязательный набор интеграционных тестов

1. **Smoke API**
   - Подъём приложения в тестовом хосте (`WebApplicationFactory`/`TestServer`).
   - Проверка, что `/metrics` отдаёт 200.
   - Проверка, что основные контроллеры мессенджеров возвращают корректный статус и JSON-контракт.

2. **Контракт с сервисным слоем**
   - Сквозной тест: HTTP запрос → контроллер → `ConversationService`/`DialogflowService` → ответ DTO.
   - Проверка ключевых веток: новый пользователь, повторный пользователь, reset contexts.

3. **Ошибки и middleware**
   - Интеграционный тест на обработку исключений (`ExceptionsMiddleware`) и формат ответа на ошибку.

4. **Набор контрактных snapshot/assert тестов**
   - Закрепить структуру JSON-ответов для Яндекс/Маруся/Сбер, чтобы быстро видеть регресс при обновлении пакетов сериализации и ASP.NET.

### 3.2 Стратегия mock Dialogflow (рекомендуется внедрить до апдейтов)

Чтобы тесты были детерминированными и не зависели от GCP:

- Добавить тестовый двойник для `SessionsClient` и `ContextsClient` (через текущие `ScopesSelector<T>`).
- Вынести фабрику клиентов в интерфейсы (`IDialogflowSessionsClient`, `IDialogflowContextsClient`) и инжектить через DI.
- Для интеграционных тестов подменять реализацию на in-memory fake:
  - заранее заданный `DetectIntentResponse` по входному тексту;
  - проверка, какие контексты создавались через `CreateContextAsync`.
- Покрыть сценарии:
  - `/start` → event `Welcome`;
  - пустой текст + `IsOldUser=true` → `EasyWelcome`;
  - слишком длинный текст обрезается до 30 символов;
  - передача required contexts в `QueryParams.Contexts`.

## 4) Пошаговый план обновления зависимостей

### Шаг A — Подготовка

1. Зафиксировать baseline:
   - прогон существующих unit-тестов;
   - прогон новых интеграционных тестов;
   - сохранение артефактов (test results).
2. Включить централизованное управление версиями пакетов (`Directory.Packages.props`) — опционально, но сильно упростит массовое обновление.

### Шаг B — Безопасные обновления патч/минор (без смены TFM)

1. Обновить `Microsoft.Extensions.*`, тестовый стек (`Microsoft.NET.Test.Sdk`, `NUnit`, `Moq`) и observability-пакеты.
2. Прогнать тесты.
3. Исправить предупреждения/обратную совместимость по одному пакету за PR.

### Шаг C — Обновление фреймворка (основной переход)

Рекомендуемый целевой фреймворк: **`net8.0` (LTS)**.

1. Обновить TFM по проектам `net6.0` → `net8.0`.
2. `FillInTheTextBot.Models` оставить на `netstandard2.0` (или перевести на `net8.0`, если библиотека не нужна внешним потребителям на старом рантайме).
3. Пересмотреть пакетные версии под `net8`:
   - `Microsoft.AspNetCore.Mvc.NewtonsoftJson`
   - `OpenTracing/Jaeger` (возможна миграция на OpenTelemetry)
   - `prometheus-net*`
   - `Google.Cloud.Dialogflow.V2`
4. Прогнать полный набор тестов + ручной smoke основных endpoint.

### Шаг D — Оптимизация после миграции

1. Убрать устаревшие API и предупреждения компилятора.
2. Проверить логирование/метрики/трейсинг в рантайме.
3. Добавить проверку обновлений в CI (например, Dependabot/Renovate).

## 5) Риски и как их закрыть

- **Риск:** изменение поведения сериализации JSON после апдейта ASP.NET.
  - **Контроль:** контрактные интеграционные тесты + snapshot.
- **Риск:** поломка интеграции Dialogflow SDK.
  - **Контроль:** mock/fake слой + отдельные e2e тесты в staging с реальным GCP.
- **Риск:** сломанные middleware/порядок pipeline.
  - **Контроль:** интеграционные тесты на статус-коды, исключения, `/metrics`.

## 6) Рекомендуемый порядок PR

1. PR-1: Интеграционные тесты + mock/fake Dialogflow слой (без апдейта пакетов).
2. PR-2..N: Серия небольших PR на обновление зависимостей внутри `net6.0`.
3. Финальный PR: миграция на `net8.0` + точечные правки совместимости.
4. Пост-миграция PR: cleanup, CI-автоматизация обновлений, документация.

## 7) Definition of Done

- Есть интеграционные тесты для ключевых пользовательских сценариев.
- Все тесты стабильны и проходят до и после обновлений.
- Приложение собирается и запускается на `net8.0`.
- Для Dialogflow есть воспроизводимый тестовый mock/fake контур.
- Проверены и задокументированы изменения из ветки `vibe-coding` (когда ветка доступна в remote).
